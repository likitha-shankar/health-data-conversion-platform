Supplement A — Canonical Internal Model (CIM)
Version: 1.0  |  Status: Required before Week 1 build  |  Owner: Platform Architecture

A.1  Purpose and Design Rationale
The Canonical Internal Model (CIM) is the single shared data representation that every specialist agent maps its source data into, and that every output serializer reads from to produce the target format. It is the universal contract between all agents and all outputs. Without a locked CIM definition, agents and serializers will drift independently and produce incompatible intermediate representations that cannot be debugged or audited consistently.

The CIM is a FHIR R4 superset. It uses FHIR R4 resource structures as its base because FHIR R4 is the most widely adopted modern clinical data model, and because using it as the CIM means the FHIR R4 output serializer is lossless by construction. Non-FHIR targets (OMOP, SNOMED records, LOINC lab output, flat JSON) have defined mapping tables from CIM fields to their target schemas.

The superset extensions added on top of FHIR R4 are: agent provenance fields (which agent version produced each field), confidence scores per field (from the terminology layer), NLP extraction metadata (when a field was produced by the NLP pipeline), and a ConflictedValue wrapper (when two source fields produced different values for the same CIM field).

A.11  ConflictedValue Wrapper
When two source fields map to the same CIM path with different values, neither is silently dropped. Both are preserved in a ConflictedValue wrapper and the CIM field is left as null until the conflict is resolved by the validation layer or human review.

conflicted_values: [
  {
    "cim_path": "$.patient.birth_date",
    "source_a": { "value": "1980-12-05", "source_path": "PID.7", "source_system": "epic" },
    "source_b": { "value": "1980-12-15", "source_path": "demographics.dob", "source_system": "lab_feed" },
    "resolution_status": "UNRESOLVED | HUMAN_RESOLVED | AUTO_RESOLVED",
    "resolved_value": null,
    "resolved_by": null,
    "resolved_at": null
  }
]

A.12  CIM Versioning Policy
The CIM schema is versioned using semantic versioning. Patch versions (1.0.x) may add optional fields. Minor versions (1.x.0) may add required fields but must include a migration guide and a grace period during which old agents producing the previous minor version are accepted. Major versions (x.0.0) break backward compatibility and require all agents to be updated before promotion.

✓
No agent may be promoted to production if its output does not validate against the current active CIM version. The CIM JSON Schema is published as a versioned artifact in the platform repository and is the single source of truth for all agent output contracts.


Supplement B — PHI Handling Specification
Version: 1.0  |  Status: Required before any patient data is processed  |  Owner: Security + Clinical Informatics

B.1  Scope and Regulatory Basis
This specification defines how Protected Health Information (PHI) is handled throughout the conversion pipeline. It covers: what constitutes PHI in this system, where PHI may and may not appear, what the PHI preprocessing gate must do before any record is processed, how PHI is handled in logs and audit records, and what constitutes a PHI gate pass versus fail.

The regulatory basis is HIPAA's Technical Safeguard requirements under 45 CFR §164.312, which require access controls, audit controls, integrity controls, and transmission security for all ePHI. State laws (e.g. CCPA, SHIELD Act) may impose additional obligations. This specification establishes the minimum floor; tenant-specific configurations may add stricter controls but never relax them.

B.3  PHI Preprocessing Gate
The PHI preprocessing gate is the first step after ingestion and before any parsing or mapping occurs. It runs as a synchronous blocking check. No record may proceed to the parsing layer until the gate produces a deterministic PASS or FAIL result. An ambiguous or timeout result is treated as FAIL.

B.3.1  Gate Components

Component
What It Does
Regex PHI Scanner
Pattern-matches against SSN formats, MRN patterns (configurable per tenant), phone number formats, email addresses, and common date-of-birth patterns appearing outside expected fields
NER PHI Detector
Microsoft Presidio (open source) runs NER-based detection for names, locations, and identifiers appearing in free-text fields or outside their expected positions
Structural Position Checker
For structured formats (HL7, FHIR), verifies that PHI appears only in expected fields. PHI detected outside expected fields (e.g. SSN in a clinical note field) raises a high-risk flag
Biometric / Image Detector
Detects base64-encoded image data or biometric patterns in fields where they should not appear; triggers immediate halt

B.4  PHI in Logs and Audit Records
🛑
Raw PHI values must never appear in any log line, metric label, trace span name, or error message. Violations of this rule constitute a potential HIPAA breach and must be treated as a P0 security incident.

The following rules apply to all log output, audit records, error payloads, and observability data:

Rule
Implementation
No raw identifiers in logs
Patient names, MRNs, SSNs, DOBs, phone numbers, and emails are replaced with tokenized references (e.g. patient_token: ptk_7f3a2b1c). The token is reversible only via the PHI Vault service with an authorized access request.
No raw clinical values in error messages
Error messages reference field paths and value hashes, not values. Correct: 'source_path: PID.7, value_hash: sha256:abc123'. Wrong: 'source_path: PID.7, value: 1980-12-05'.
Audit trail stores value hashes by default
The audit_field_transform table stores SHA-256 hashes of source and target values by default. Full values are stored only in the PHI Vault with a reference pointer in the audit record. Debug replay can retrieve full values from the vault with authorization.
Stack traces must not contain field values
Exception handlers must scrub any CIM record fields from stack traces before logging. Use a dedicated CIM scrubber middleware on all exception handlers.
Distributed traces contain no PHI
Jaeger/OpenTelemetry span attributes must only contain: trace_id, span_id, ingestion_id, agent_id, layer, status. No clinical values, no patient identifiers.

B.5  PHI in Storage
Storage Layer
PHI Handling
Raw record archive (S3)
Encrypted at rest with AES-256; bucket policy prevents public access; KMS key rotation every 90 days; access logged to CloudTrail. Original bytes retained for audit replay. Lifecycle policy: retain for 7 years, then secure deletion.

Encrypted at rest; value fields store hashes only by default; full values in PHI Vault. Row-level security by tenant_id. Append-only — no UPDATE or DELETE permissions granted to application roles.

Output store (OMOP, FHIR repo)

PHI Vault
HashiCorp Vault or AWS Secrets Manager with fine-grained policies; access to full PHI values requires an authorized access request logged to the HIPAA audit log; emergency break-glass access requires two-person authorization.

B.6  Breach Response Protocol
If PHI is detected in a log, error message, trace, or any unauthorized storage location, the following steps apply immediately:

Step
Action
1
Contain: disable the log stream or storage location where PHI was found. Do not delete — preserve evidence.
2
Notify: alert the security lead and HIPAA Privacy Officer within 1 hour of detection.
3
Assess: determine scope — how many records, which PHI elements, how long the exposure existed.
4
Remediate: identify and fix the code path that allowed PHI to escape. Add a regression test that would have caught it.
5
Report: if the breach involves more than 500 individuals, notify HHS within 60 days. Affected individuals notified within 60 days regardless of count.


Supplement C — Format Detection Engine Specification
Version: 1.0  |  Status: Required before orchestrator build  |  Owner: Platform Architecture

C.1  Purpose
The format detection engine is the first decision point in the orchestrator. Every downstream routing decision — which specialist agent runs, which parser is invoked, which error codes apply — depends on its output. An unspecified or poorly calibrated detection engine is a hidden source of misroutes, which in this system means a wrong agent processes a record and potentially produces clinically incorrect output. This specification defines exactly how detection works, how confidence scores are calculated, and what the thresholds and tie-breaking rules are.

C.2  Detection Architecture
Format detection runs as a three-stage pipeline. Each stage adds evidence to a shared evidence accumulator. The final confidence score for each candidate format is a weighted sum of evidence from all three stages. Detection is always completed before any agent is invoked — it runs in the orchestrator, not inside any specialist agent.

Stage
Name
What It Does
1
Byte-level sniffer
Inspects the first 512 bytes for byte order marks, delimiter patterns, framing characters, and magic byte sequences. Runs in <5ms. Produces a set of candidate formats with initial weights.
2
Structural probe
Attempts lightweight structural parsing of the top 2 candidates from Stage 1. Checks for required structural landmarks (e.g. MSH segment, JSON root object, XML namespace, PDF header). Does not attempt full parsing. Runs in <50ms.
3
Agent self-declaration
Each registered agent's supportsSource() method is called with the Stage 1 + 2 evidence. Agents return a SupportDecision with a score and evidence. The orchestrator aggregates agent self-declarations as a third evidence source.

C.4  Confidence Score Calculation
The raw confidence score for each candidate format is computed as a bounded weighted sum of all applicable positive and negative signals. Scores are normalized to the range [0, 1]. The detection engine produces a ranked list of candidates with their scores.

raw_score(format) = Σ(positive_weights) + Σ(negative_weights)
normalized_score(format) = max(0, min(1, raw_score(format)))

C.6  Tie-Breaking Rules
When two formats score within 0.20 of each other (FORMAT_AMBIGUOUS), the following ordered tie-breaking rules are applied before routing to human review. If any rule produces a clear winner, routing proceeds with a note that tie-breaking was applied:

Rule
Condition
Resolution
1
Sender profile has declared an expected format for this source_system_id
Use the declared format if it is one of the tied candidates and its score is ≥ 0.60. Record tie_broken_by: SENDER_PROFILE.
2
One candidate has a byte-level signal with weight ≥ 0.65 and the other does not
Prefer the candidate with the strong byte-level signal. Byte signatures are more reliable than structural probes. Record tie_broken_by: BYTE_SIGNAL_DOMINANCE.
3
Transport channel strongly implies a format (e.g. MLLP always means HL7 v2)
Route to the transport-implied format if its score is ≥ 0.60. Record tie_broken_by: TRANSPORT_CHANNEL.
4
No rule resolves the tie
Route to FORMAT_AMBIGUOUS human review. No automatic routing. The review packet includes all signals and scores for each candidate.

C.7  Human Review Resolution Flow
When a record is routed to the human review queue due to FORMAT_AMBIGUOUS or FORMAT_UNKNOWN, the following flow applies:

Step
Action
1
Reviewer sees: original bytes (first 1KB displayed), all candidate formats with scores, all contributing evidence signals, and tie-breaking rules that were attempted.
2
Reviewer selects the correct format and provides a confidence justification note.
3
If the resolved format comes from a known sender (source_system_id is set), the reviewer may optionally update the sender profile to pre-declare this format for future records from the same source, preventing future FORMAT_AMBIGUOUS routing for this sender.
4
Record is re-injected into the pipeline with forced_source_format and reviewer_id set. The orchestrator bypasses detection and routes directly to the specified agent, recording that routing was human-resolved.
5
Every human-resolved detection case is reviewed weekly. If the same source_system_id produces repeated FORMAT_AMBIGUOUS events, a sender profile update is mandatory — repeated ambiguity from a known sender is a configuration gap, not a routing edge case.

The format detection engine is a critical path component. Any regression in detection accuracy directly causes misroutes, which are the most dangerous failure mode in the entire platform. Every change to detection signal weights or thresholds must be evaluated against the full golden test set before deployment.
